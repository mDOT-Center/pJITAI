Model Settings / Proximal Outcome Attributes

  • Explanation
  • HeartSteps Example

We use the attributes of the proximal outcome to check if the API receives inappropriate values from your mobile app. Please read [tutorial] for more details.

If you're reading the HeartSteps demo, the HeartSteps numbers look weird because in HeartSteps the proximal outcome (the 30-minute step count after each decision point) is log-transformed. (See HeartSteps Example for more explanation.)

You determine whether your proximal outcome needs to be transformed. Please note that if your proximal outcome needs to be transformed, the transformation needs to be implemented in the mobile application; the algorithm will not do the transformation. Hence, if your proximal outcome need not be transformed, we are asking for the minimum and maximum values of the proximal outcome. If there is transformation, we are asking for the minimum and maximum values after the transformation.

In this version of pJITAI, we assume the proximal outcome is continuous. In the future, categorical proximal outcomes will be incorporated.

In HeartSteps, the proximal outcome, the 30-minute step count after each decision point, is right-skewed. Thus, the HeartSteps team decided to log-transform the 30-minute step count after each decision point so that the normal approximation used in the linear regression is more accurate. To avoid minus infinity (recall that loge(0) becomes minus infinity), we added 0.5 to the step count before the transformation, i.e., loge(next 30-minute step count+0.5).

Our assumption was that people walk between 0 and 3000 steps in the next 30 minutes. Hence, the minimum value of the (log) 30-minute step count is loge(0+0.5)=-0.69. The maximum value of the (log) 30-minute step count is loge(3000+0.5)=8.0. These values are continuous.
